### 步骤2: 观察输出并分析

| 如果看到... | 含义 | 处理 |
|-----------|------|------|
| 没有具体的块号 | 没有检测到泄漏 | ✅ 良好，继续监控 |
| `{1234} normal block at...` | 检测到泄漏块 | 🔧 需要修复 |
| `Normal块数: 0` | 没有分配任何内存 | ✅ 良好 |
| `Normal块大小: 很大` | 分配了大量内存 | ⚠️ 需要优化 |

### 步骤3: 如果找到泄漏块

假设看到：
```
{12345} normal block at 0x00A1B2C3, 256 bytes long.
        D:\Project\file.cpp(123)
```

**做这些**:

1. **记下块号**: 12345

2. **在InitInstance中添加**:
```cpp
#ifdef _DEBUG
    // 内存检测设置...
    _CrtSetBreakAlloc(12345);  // ← 添加这行！
#endif
```

3. **重新运行**:
   - F5启动调试
   - 程序会在分配该块时自动中断
   - 查看调用栈确定分配位置
   - 查找对应的释放代码

## 💡 常见情况和解决方案

### 问题1: 看到泄漏块，但找不到释放代码

**原因**: 对象被创建后没有正确释放

**解决方案**:
```cpp
// ❌ 错误
MyObject* obj = new MyObject();
obj->DoSomething();
// 忘记了 delete obj;

// ✅ 正确 - 方案1: 手动释放
MyObject* obj = new MyObject();
obj->DoSomething();
delete obj;  // ← 添加这行

// ✅ 正确 - 方案2: 智能指针（推荐）
auto obj = std::make_unique<MyObject>();
obj->DoSomething();
// 自动释放，不需要delete
```

### 问题2: MFC对象的泄漏

```cpp
// CString, CMenu, CWnd 等通常不需要手动释放
CString str = _T("Hello");  // ✅ 自动释放
CMenu menu;
menu.CreateMenu();          // 后续在WM_DESTROY中通过MFC释放
```

**不用担心** - MFC会在合适的时机自动释放这些对象。

### 问题3: 第三方库的泄漏

如果泄漏来自 `libsparkusb.dll` 或其他DLL：

**检查方法**:
```cpp
// 在输出中查找文件路径
{12345} normal block at 0x..., 256 bytes long.
        libsparkusb.dll!....  ← 来自第三方库
```

**处理**:
- 联系库的维护者报告问题
- 或检查库的使用文档是否有特殊的清理函数
